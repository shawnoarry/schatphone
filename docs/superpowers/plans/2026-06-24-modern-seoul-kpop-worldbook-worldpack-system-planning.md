# 现代首尔 K-pop 世界书与世界包系统规划清单

Updated: 2026-06-24

Status: `PLANNING_DRAFT`

## 目的

这份清单用于治理“现代首尔 K-pop 娱乐圈”世界观文档包、Book/WorldBook 装载、World Pack 拓展、联系人模板、服务号、日程地点和事件链路之间的关系。

目标不是继续堆百科，而是确认：

1. 已经有什么；
2. 哪些内容重复；
3. 哪些缺口应该补在现有文稿里；
4. 哪些不该写成百科，而应该做成人设模板、世界包、服务号、日程类型、地点库或事件种子；
5. 后续替换装载时如何避免上下文膨胀。

## 当前系统事实

### Book 文本库

Book 是长文本素材库。当前正式分类只有三类：

- 世界观
- 百科
- 世界规则

Book 文本可以整篇或按章节链接到 WorldBook。Book 本身保存文本，WorldBook 负责启用和进入上下文。

当前代码仍兼容旧名字，例如 `knowledge_note`、`reference_note`、`rule_set` 等，但用户可见分类已经应统一成“世界观 / 百科 / 世界规则”。

### WorldBook 上下文

WorldBook 会把启用的 Book 文本送入上下文，但预算很小：

- 启用 Book 文本总预算约 2400 字；
- 单条启用文本最多约 1200 字；
- 角色绑定百科最多约 8 条。

因此正式装载不能把所有百科全文同时启用。长百科应作为可选素材或分章选择；真正进入上下文的正文需要短、准、有效。

WorldBook 当前会影响：

- Chat；
- Map；
- Calendar；
- Event Runtime。

Contacts 不直接消费 prompt 上下文。Contacts 使用 WorldBook 提供的世界专属人设模板字段，由用户在联系人里填写具体值。

### World Pack

World Pack 当前不是“默认世界观全文”。它更像当前世界的拓展能力包，能携带：

- 关联百科 ID；
- 世界角色模板 ID；
- Book 文本链接 ID；
- 应用入口绑定；
- 服务号模板；
- 世界术语；
- 兼容性标签；
- 货币；
- 关系分类扩展。

World Pack app binding 当前可用原型包括：

- publication_feed：发布/频道入口，通常指向 Chat；
- reservation：预约/日程入口，通常指向 Calendar；
- transit：交通/路线入口，通常指向 Map；
- marketplace：市场入口，通常指向 Shopping；
- dispatch：调度入口，通常指向 Food Delivery；
- subscription：订阅入口。

World Pack app binding 默认只提供入口、命名、上下文和轻量 UX 语言，不拥有目标 App 的真实业务记录。

### Chat 与服务号

Chat 会读取 WorldBook 上下文并拼入 AI prompt。Chat 还负责服务号/公众号的用户加入、编辑、消息线程和源模块通知展示。

WorldBook 当前只显示某个世界包有哪些服务号候选；真正创建和加入服务号在 Chat 的 Services 管理区。

K-pop 世界特别适合使用服务号，例如：

- 公司公告号；
- 团体官方号；
- 粉丝俱乐部号；
- 行程通知号；
- 打歌节目通知号；
- 票务/签售结果号；
- 品牌活动通知号；
- 周边/物流通知号。

### Contacts 与人设模板

Contacts 拥有具体人物档案。WorldBook 可以准备当前世界的人设模板字段，Contacts 保存每个角色、Self Profile、NPC 的具体值。

当前支持的模板字段类型包括：

- 短文本；
- 长文本；
- 单选；
- 多标签；
- 人物引用。

K-pop 已出道艺人的沉浸体验需要优先落到 Contacts 模板，而不是写成百科。

### Calendar、Map、Reminders

Calendar 拥有已确认日程。K-pop 日程类型应进入 Calendar 语境，而不是只存在百科里。

Map 拥有地点、路线、移动和旅行连续性。K-pop 的公司楼、沙龙、电视台、机场、酒店、场馆、品牌现场适合进入 Map 语境。

Reminders 拥有跨模块 cue，例如经纪人提醒、开票提醒、签售截止、直播提醒、移动前确认。

### Event Runtime 与 World Hub

事件运行时可以生成或审核低风险事件，例如问候、提醒、服务号通知、日程相关动态。高影响关系、冲突、封锁、拒绝、恢复等需要 World Hub 审核。

事件运行时不适合直接读取大段百科。它更需要短标签、事件类型、触发条件、冷却、来源说明和可审查的事件种子。

## 刚删除草稿的审计结论

被删除的“已出道艺人日程与生活状态百科”不是完全没有新增，但它不适合作为新的大百科保留。

它的内容大部分已经散落在现有大稿中：

- 回归准备、录音、编舞、MV、打歌、综艺、签售、演唱会、工作人员：已有于“作品、舞台与现场生产百科”。
- 新人期、上升期、solo、团体空窗、海外、品牌、续约、转型：已有于“行业与事业运转百科”。
- 论坛、官方号、粉丝动态、舆情、服务提醒：已有于“粉丝、平台与舆情生态百科”。
- 作息、车上、机场、酒店、身体状态、关系边界、私聊状态：已有于“城市生活、日常状态与关系网络百科”。

它真正提供的新增价值是“已出道艺人视角的组合方式”，更适合拆到以下位置：

1. 已出道艺人人设模板；
2. 日程类型与状态标签；
3. 服务号候选；
4. 事件种子；
5. 已有百科的局部精修。

因此处理原则是：不新增“已出道艺人日程百科”类别；把新增价值归位到更合适的系统载体。

## 内容缺口清单

### A. 需要继续精修的现有百科

#### 行业与事业运转百科

当前已有完整骨架。需要补强的是“已出道艺人视角”的表达，而不是新增大类。

可补内容：

- 新人期、上升期、成熟期、顶级艺人的生活差异；
- 团体空窗期的个人活动、粉丝焦虑和公司排期；
- solo 准备与团体活动之间的关系；
- 品牌资源如何影响日常行程和公开边界；
- 低曝光期、续约期、转型期如何改变聊天语气与对外露出。

输出改善场景：

- Chat 能根据角色事业阶段调整压力、语气和话题；
- Contacts 模板能有更准确的事业阶段字段；
- Calendar 可以区分回归期、巡演期、空窗期、品牌期。

#### 作品、舞台与现场生产百科

当前已经很细。需要避免继续扩写制作知识，转向“艺人身处其中时的状态”。

可补内容：

- 打歌日程从沙龙、到台、预录、采访、下班到复盘的状态链；
- 综艺录制前后：台本确认、禁谈话题、录后担心剪辑；
- 签售/video call 的重复营业、记忆感和疲惫；
- 演唱会结束后的兴奋、脱力、复盘和下一站移动。

输出改善场景：

- Chat 私聊能自然出现“刚下预录”“还在待机室”“PD 让再补一遍”；
- Calendar 能形成具体通告类型；
- Event Runtime 可以生成低风险现场前后事件。

#### 粉丝、平台与舆情生态百科

当前覆盖面较全。需要把“外部声音如何影响角色”写得更明确。

可补内容：

- 轻度动态：饭拍、repo、机场图、直播切片、品牌图；
- 中高压动态：热帖、营销号标题、黑热搜、站子表态、品牌观望；
- 官方号和服务提醒的语气边界；
- 粉丝解读与事实之间的距离。

输出改善场景：

- 论坛/动态类输出更像外部生态；
- Chat 里角色会因外部声音改变情绪；
- 服务号和官方号更容易有稳定文风。

#### 城市生活、日常状态与关系网络百科

当前覆盖首尔生活、地点、作息、关系边界。需要把“艺人身份导致的普通生活变形”再压实。

可补内容：

- 公司车、沙龙、机场、酒店、宿舍这些地点对聊天状态的影响；
- 公开身份如何影响发照片、吃饭、见朋友、看展、去医院；
- 作息标签如何改变角色活跃时段和语气，而不是直接创造实时回复；
- 身体状态如何影响短句、慢回、沉默、低电量。

输出改善场景：

- Chat 不再像百科解释，而像角色真的在某个状态里；
- Map 地点能有更具体的语境；
- Contacts 的作息/生活状态字段更有实际意义。

#### 行业名人与功能性角色百科

当前已经开始承接功能 NPC。需要从“名人列表”再推进到“功能角色如何参与事件”。

可补内容：

- MC、PD、作家、制作人、A&R、品牌 PR、记者、站姐、粉丝运营在事件中的作用；
- 临时 NPC 与联系人角色的区别；
- 哪些 NPC 适合进入通讯录，哪些只适合事件中出现；
- 真实名人、虚构功能 NPC、用户联系人之间的边界。

输出改善场景：

- Event Runtime 可以生成更像行业生态的临时角色；
- Contacts 可在用户确认后新增功能性联系人；
- Chat 能自然出现节目组、品牌方、经纪团队等非亲密关系消息。

## 上下文策略

### 默认启用建议

默认不应同时启用所有百科。

- 主世界观；
- 世界规则；
