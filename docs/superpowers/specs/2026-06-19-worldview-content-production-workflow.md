# Worldview Content Production Workflow

Updated: 2026-06-20

Status: `WORKFLOW_ACCEPTED`

适用范围：所有内置世界观、百科、语态/文风补丁、世界规则、人设模板和内部参考文稿。

## 1. 核心原则

世界观正文、百科正文和语态补丁正文是给上下文使用的设定素材。它们应写成世界内信息：这个世界有什么、如何运转、角色会遇到什么、不同场景如何形成压力、关系与表达方式。

主世界观、百科正文和语态补丁不得混入后台程序说明、文档调用说明、回复调度规则或开发流程说明。表达风格、称谓习惯、平台语态和世界内说话方式可以进入正文；触发频率、调用时机、返回格式、模块写入和业务状态变更只能进入“世界规则”、代码或系统提示词。内部审核、来源、参考、事实校订和装载计划只进入内部参考或计划文档。

核心边界：表达风格可以写，执行机制不能写。世界内说话方式可以写，后台如何调用不能写。

机制优先由代码稳定实现，不让用户通过上下文承担程序控制。上下文只写能真实改善生成结果的内容；每段装载正文都必须能说明它会让哪个 AI 输出场景变好，以及具体变好在哪里。不能改善角色回复、论坛内容、服务号消息、公告、日程/地点事件、世界 App 外壳或人设字段建议的文字，不进入装载正文。

所有文稿必须优先保留用户可改写空间。真实实体、行业机制和成员坐标作为参照，不作为限制条件。用户写明的 AU 名册、角色关系、履历、地点、日程和事件优先于默认资料。

## 2. 文稿分类

开始写作前必须先判断文稿类型。

### 主世界观

用途：定义世界整体骨架。

正文内容应包括：时代、地点、行业结构、主要实体类型、角色范围、冲突来源、社交生态、商业机制、日程地点、AU 可变性。

正文不得写：回复方式、自动消息、加载方式、后台分类名、系统如何选择文本、文档如何被调用。

### 百科

用途：补充可进入上下文的专业语境、术语、实体、生活方式、成员坐标、舆情机制。

正文内容应包括：条目定义、场景作用、使用边界、关键词、不同身份或场景的差异。

百科应服务角色对话和系统生态，不应写成面向用户的教学说明。每个条目都要能解释世界内会出现什么语气、行动、压力、关系、地点、日程或讨论。

### 语态 / 文风补丁

用途：补充特定世界、平台、身份或社群的表达方式。

正文内容应包括：称谓习惯、敬语/半语、私聊语气、论坛语气、服务号或官方号口吻、粉丝社区表达、公告体、直播评论感、跨语言直译感、身份差异带来的措辞变化。

语态补丁可以直接影响角色、论坛、服务号、公告和社交内容的表达风格。语态补丁不得控制触发时机、主动消息频率、回复条数、JSON 格式、图片/语音权限、模块写入或业务状态变化。

### 世界规则

用途：补充回复媒介、时间系统、线上线下边界、主动性边界、通知和日程事实边界。

世界规则可以写功能机制，因为它本身就是规则文档。回复方式、短消息模式、真实时间、自动表现边界、位置和日程确认等内容只允许放在世界规则，不放进主世界观或百科。

### 人设模板

用途：提供角色资料填写结构。

模板应字段化、可编辑、尽量角色中立。绑定世界观的人设模板需要明确其适用世界和启用条件；通用模板不绑定特定世界观。

### 内部参考

用途：审核、事实校订、实体候选、来源记录、装载计划、批次索引。

内部参考可以包含表格、来源、链接、候选清单、审计备注和待确认问题。内部参考不得直接作为加载正文使用。

## 3. 正文禁入项

主世界观和百科正文禁止出现以下内容：

- 后台程序名：`Book`、`WorldBook`、`Contacts`、`Calendar`、`Map` 等。
- 后台动作词：调用、装载、加载、索引、系统选择、上下文拼接、代码集成。
- 模型指令词：`AI 应`、`AI 可以`、`模型必须`、`回复应`、`生成时`。
- 功能规则：自动回复、主动消息频率、线上线下切换规则、实时调度、系统通知触发。
- 来源污染：消息来源、资料来源、参考资料、参考链接、URL、事实审核轨迹。
- 展示污染：Markdown 表格、过长审计备注、给用户解释本文件用途的说明。

允许在世界规则中出现回复媒介、时间、日程、地点、通知和主动性边界。允许在内部参考中出现来源、表格、审核轨迹和装载计划。

## 4. 设定表达转换

写作时必须把后台视角转换成世界内表达。

错误：`AI 可以用公司资源差异塑造角色压力。`

正确：`公司资源差异会塑造角色的压力、机会和说话方式。`

错误：`本世界观不预设剧情入口。`

正确：`这个世界支持练习生、已出道 idol、制作人、经纪人、粉丝站、记者、品牌方和普通首尔年轻人等多种角色位置。`

错误：`Book / WorldBook 增补文本负责补足专业语境。`

正确：`补充设定可以扩展专业语境、实体范围、关系网、事件线和 AU 名册。`

错误：`Chat 中使用术语时保持短消息语气。`

正确：`手机私聊中术语使用较少，语气更短、更贴近角色关系。`

错误：`AI 应直接接受用户写明的 AU 名册。`

正确：`AU 名册、履历和关系以用户写明版本为准。`

## 5. 写作流程

### 5.1 开始前确认

写作前必须先列出本轮文稿要解决的问题。

至少确认：

- 文稿类型：主世界观、百科、语态/文风补丁、世界规则、人设模板或内部参考。
- 进入范围：只做草稿、准备替换已装载正文，或准备后续集成。
- 需要用户决定的点：真实实体范围、AU 自由度、术语范围、百科类目、事实细节、敏感边界。
- 本轮不做的事：代码集成、索引更新、事实校订、装载替换或其它范围外工作。

拿不准的问题必须问用户。一次只问一个关键问题，避免一次抛出过多分支。

### 5.2 先写计划

新世界观或大批量百科开始前，先写计划文档。计划文档应包含：

- 当前已知设定。
- 待确认问题。
- 拟写文稿清单。
- 每份文稿的世界内作用。
- 每份文稿影响的是世界事实、行为逻辑、语言风格、平台语态还是系统机制。
- 每份文稿预计改善的输出场景：角色回复、论坛内容、服务号消息、公告、日程/地点事件、世界 App 外壳、人设字段建议或其它明确场景。
- 哪些需求应由代码、系统提示词或世界规则解决，不应交给上下文。
- 哪些内容只放内部参考。
- 哪些内容禁止进入加载正文。

用户确认计划后再写正文。

### 5.3 草稿写作

草稿正文优先使用直接陈述句。每段必须贡献具体设定信息，例如实体、机制、场景、身份、关系、地点、日程、舆情、商业规则或生活状态。

不得为了“更戏剧化”锁定用户角色起点。世界观负责框定整体世界，用户人设和补充设定负责决定具体身份、地位、关系、事件和剧情方向。

百科类目应合并同类项，避免碎成过多只解释概念的小词条。每个百科都要回答：这个内容进入上下文后能帮助角色、论坛、服务号、日程、地点或粉丝生态表现什么？

语态/文风补丁应写具体表达特征，不写抽象审美口号。每个补丁都要回答：这种语气由什么身份、平台、关系、年龄、行业位置或舆情压力形成？它适合出现在私聊、论坛、官方消息、粉丝社区、直播评论还是公告里？它不负责决定内容何时生成。

### 5.4 用户审核

提交用户审核时必须说明：

- 本轮新增或修改了哪些文稿。
- 每份文稿的世界内作用。
- 是否涉及真实实体、成员资料、年龄、舆情、商业规则或其它需要事实校订的内容。
- 哪些内容仍只是草稿，哪些准备后续替换加载正文。

用户指出方向错误时，先修正流程和文体边界，再继续写内容。

### 5.5 批次完成后再更新索引

同一批百科或世界观文稿未完成前，不反复更新索引。索引只在一批文稿结构稳定后更新，避免重复维护和错误传播。

### 5.6 已装载文稿替换流程

当用户要求“检查装载文档”“替换已装载正文”“把草稿装入 Book / WorldBook”时，先做替换审计，不直接改代码或索引。

第一步：确认当前装载源。

- 查找内置 Book 文稿注册位置。当前项目的内置 Book 文稿由 `src/lib/built-in-book-assets.js` 注册。
- 记录每个已装载文稿的标题、分类、asset id、sourcePath、正文截取起点、正文截取终点。
- 判断该文稿只是内置可选源，还是已经通过用户保存的来源链接进入当前世界上下文。
- 如果正文截取起点或终点与草稿标题不匹配，先记录为替换风险。例如草稿写 `## 正文`，注册处却按 `规则定位` 截取，就可能把草稿头部一起带入正式正文。

第二步：确认草稿与装载的重复关系。

- 列出当前已装载文稿。
- 列出同主题草稿。
- 标记关系：完全替代、拆分新增、压缩总览、只作内部参考、暂不装载。
- 同一主题不要让多份正文同时承担同一种上下文职责。
- 如果新草稿把旧装载文稿拆成多个主题，旧文稿应压缩为总览，新文稿承担细节。

推荐判断：

- 主世界观：只承担世界骨架，不承载百科细节。
- 世界规则：只承担媒介、时间、线上线下、事实确认和通知边界。
- 行业机制百科：适合做总览，不适合无限扩写成所有 K-pop 细节合集。
- 术语百科：适合做词库、语气和简称，不承担完整粉丝生态。
- 生活方式百科：承担年轻人生活习惯、作息、社交和城市日常，不承担饭圈组织或公司运营。
- 实体坐标百科：承担公司、团体、节目、平台和成员坐标，不承担事件和争议事实。

第三步：决定替换方式。

- 语义相同、只是正文升级：保留原 asset id，更新对应 sourcePath 或正文内容。这样用户已有来源链接还能识别同一个内置源，并通过变更确认流程刷新快照。
- 旧文稿拆出新主题：旧 asset id 保留给压缩后的总览；新主题使用新 asset id、新标题和新 sourcePath。
- 旧文稿语义错误或不再适合展示：不要直接删除。先保留兼容入口或迁移说明，确认没有测试、旧存档或用户链接依赖后再移除。
- 草稿仍在审核：不要接入内置注册。保持在 `docs/superpowers/content/` 作为素材层。
- 准备正式装载：优先建立干净发布稿或确认草稿已清理到可装载状态，再改内置注册。

第三步补充：合并完成后的旧稿处理。

- 旧小稿只在合并核对期间临时存在，不作为长期版本保留。
- 已完整进入大百科、且没有内置注册、测试、索引或用户链接依赖的小稿，应退出正文工作区。
- 仍被当前项目引用的小稿不得直接移除。先完成入口替换、兼容处理或注册迁移，再移除旧稿。
- 仍含有未进入大百科内容的小稿不得移除。先把遗漏内容合入对应大百科，再清理旧稿。
- 合并后的正文工作区应以少数大百科为主，旧小稿不与大百科长期并列，避免同一主题重复占用上下文和审核注意力。

第四步：发布源与草稿源分离。

长期推荐把“素材草稿”和“正式内置源”分开管理。草稿区可以保留状态、内部备注、后续校订点、审核说明；正式内置源只保留会进入上下文的干净正文。

可选方案：

- 方案 A：继续从 `docs/superpowers/content/` 读取，但只允许已清理草稿进入注册。成本低，风险是草稿与正式源边界不清。
- 方案 B：建立 `docs/superpowers/published/` 一类发布目录，内置注册只读取发布稿。成本中等，最符合当前内容生产流程。
- 方案 C：把正式文稿放到 `src/content/built-in-book/` 一类代码侧内容目录。工程边界清晰，但内容编辑门槛更高。

默认建议使用方案 B。除非用户明确要求快速试装，否则不要把仍含内部备注、来源、索引计划或待确认项的草稿直接注册为内置源。

第五步：正式正文清理。

发布稿或可装载正文必须删除：

- `Updated`、`Status`、资产类型、用途、文体规范等草稿头部。
- 内部校订备注。
- 后续校订点。
- 来源、参考、链接和事实审计轨迹。
- Markdown 表格。
- 面向开发者或后台流程的说明。

正文可保留：

- 标题。
- 世界内定位。
- 条目设定。
- 关键词。
- 世界内边界。

如果当前内置注册依赖 `startHeading` 和 `endHeading` 截取正文，必须保证标题完全匹配。替换前检查：

- 起点标题存在。
- 终点标题存在或确认无需终点。
- 截取后的正文不含草稿头部。
- 截取后的正文不含内部备注。

第六步：替换后的验证。

正式替换后至少检查：

- Book 列表能看到预期标题、分类和标签。
- WorldBook 文本选择器能按世界观、世界规则、百科分类显示新文稿。
- 新增文稿不会挤掉旧文稿，除非明确是替代。
- 已有旧 asset id 的用户链接不会断。
- 已启用来源在正文变更后能进入变更确认或刷新路径。
- Chat prompt context、thread WorldBook summary 和 runtime world context 能读取到替换后的文本。
- 测试期望同步更新，尤其是 Book store、WorldBook source picker、World interface 相关测试。

第七步：交付说明。

替换交付必须说明：

- 哪些文稿已经正式装载。
- 哪些草稿仍未装载。
- 哪些旧文稿被压缩为总览。
- 哪些新文稿使用了新 asset id。
- 是否保留了旧链接兼容。
- 是否需要用户到 WorldBook 中确认刷新已启用来源。

如果本轮只做审计或规范，不得声称已完成装载替换。

## 6. 装载正文检查

任何准备进入正式上下文的正文，都必须通过以下检查。

### 6.1 后台词扫描

主世界观、百科正文和语态补丁正文不得出现：

```text
AI
Book
WorldBook
Contacts
Chat
Calendar
Map
调用
装载
加载
索引
自动回复
主动消息
系统选择
上下文拼接
```

如确实需要表达对应概念，改成世界内语言：

- `Chat` -> `手机私聊`
- `Calendar` -> `日程记录`
- `Map` -> `地点记录`
- `Book / WorldBook 增补` -> `补充设定`
- `调用强度` -> `使用密度`
- `AI 使用` -> `场景用途`
- `回复方式` -> `表达方式` 或 `说话方式`

世界规则文件可以保留功能词，但不得把世界规则内容复制进主世界观、百科或语态补丁。

### 6.2 来源和格式扫描

准备进入上下文的正文不得出现：

```text
消息来源
资料来源
参考资料
参考来源
http://
https://
Markdown 表格
```

来源、链接和表格必须留在内部参考文件。

### 6.3 语义检查

逐段检查每段是否属于以下任一类：

- 世界事实。
- 行业机制。
- 角色身份与关系。
- 地点、日程和生活状态。
- 论坛、社媒、粉丝生态、服务号语气和平台表达。
- 术语、实体或成员坐标。
- 称谓、语气、敬语/半语、跨语言直译感和身份表达差异。
- 世界内边界。

如果一段只是在解释“本文档如何被使用”，删除或改写。

继续检查每段是否能回答：

- 它改善哪个 AI 输出场景？
- 它让输出变得更具体、更沉浸、更符合世界，还是更符合角色身份？
- 如果没有这段，输出会缺少什么可观察差异？

如果回答不出来，删除、改写为有效设定，或移入内部参考。

### 6.4 文体检查

检查是否存在：

- 高频 `不是……而是……`。
- 防御性免责声明。
- 过度文学化段落。
- 给用户讲课的百科口吻。
- 只解释概念、不能服务角色互动的条目。
- 把代码可以稳定处理的机制交给上下文承担。
- 写了看似正确但不会影响任何输出的空泛说明。

发现后改成直接、短、可编辑、关键词密集的设定句。

## 7. 事实与真实实体边界

使用真实城市、真实公司、真实团体、真实节目、真实平台、真实成员资料时，必须区分坐标与事件。

允许：

- 真实实体作为行业坐标。
- 公开身份、出道年、团体名、节目名、平台名作为语境材料。
- 用户自设角色加入现实团体、替代成员、改写名册、改变关系和事业线。

禁止：

- 把现实指控、私密事实、未证实爆料写成默认事实。
- 把虚构事件伪装成现实新闻。
- 在装载正文中加入来源、链接、审计轨迹。

需要事实校订的资料应放入内部参考，确认后再把干净正文移入百科。

## 8. 世界规则隔离

以下内容只写入世界规则：

- 手机私聊的默认媒介。
- 短消息长度和线上模式。
- 真实时间、跳时、回忆、梦境和 AU 时间线。
- 线下同场互动触发条件。
- 日程、地点和业务记录的事实确认。
- 服务号、通知、论坛和粉丝平台的边界。
- 主动消息、自动表现和作息字段的边界。
- JSON 结构、回复条数、富媒体权限、通知触发、模块写入和业务状态变化权限。

主世界观、百科和语态补丁可以写世界中存在手机私聊、日程、地点、通知、服务号和论坛；不能写它们在程序中如何触发或如何回复。

语态/文风补丁可以写手机私聊更短、论坛更碎片化、服务号更规整、公司公告更克制、粉丝社区更高密度使用术语；不能写系统必须在某个模块、某个频率、某个返回格式下生成这些内容。

当某个需求可以由代码、结构化字段、系统提示词或世界规则稳定实现时，不把该机制伪装成世界观、百科或语态补丁。世界文本只提供代码无法稳定推导的语义材料，例如行业常识、社交压力、生活状态、称谓习惯、平台语气和世界内价值判断。

## 9. 修错流程

如果用户指出文稿方向错误，立即暂停新增正文。

修错顺序：

1. 复述用户指出的边界。
2. 找出污染类型：后台词、功能规则、教学口吻、默认剧情、过度类目、事实风险或格式污染。
3. 先修正文体和流程，再继续写内容。
4. 对同批文稿做同类扫描，避免只修一处。
5. 把这次错误转化为流程规范或检查项。

## 10. 交付说明

每次世界观文稿任务结束时，交付说明应包含：

- 修改了哪些文稿。
- 本轮只做了草稿、审核、替换准备还是正式集成。
- 是否改动代码。
- 是否运行了格式或污染扫描。
- 剩余风险：事实未校订、装载未替换、用户未确认实体范围等。

不要在 final 中复述大段文稿。只说明用户需要知道的变化和下一步。
