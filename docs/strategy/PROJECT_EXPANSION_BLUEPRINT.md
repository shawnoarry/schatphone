# SchatPhone Project Expansion Blueprint

Updated: 2026-05-19

Purpose: record long-range expansion themes for SchatPhone beyond the current execution horizon.

This file is intentionally broader and looser than:

- `docs/strategy/PROJECT_ITERATION_PLAN.md`
- `docs/roadmap/TODO_ROADMAP.md`
- task-package docs under `docs/pm/`

Use it for:

- expansion themes;
- future ecosystem directions;
- "what the product may eventually grow into" discussions.

Do not use it as the current implementation order.

## 1. Product Expansion Thesis

SchatPhone should keep growing as a virtual phone ecosystem, not as a pile of isolated module demos.

Expansion should strengthen:

- continuity;
- cross-module coherence;
- believable role/world presence;
- meaningful user action;
- optional depth for users who want more system/game-like play.

Expansion should not simply mean:

- more pages;
- more toggles;
- more visible backend feeling;
- more hidden automation without review.

## 2. Long-Range Expansion Themes

### 2.1 Real-time world rhythm

Future growth can continue around:

- stronger real-time scheduling;
- restore-time settlement;
- quiet-hour behavior;
- notification-aware life rhythm;
- later background-autonomy decisions if backend complexity becomes worth it.

### 2.2 Relationship and growth systems

Future growth can continue around:

- richer relationship metrics and stage logic;
- stronger role behavior continuity;
- better memory compression/expiry/review rules;
- more world-aware continuity;
- optional deeper review/control through World Hub and future Cheats.

### 2.3 Cross-module role presence

The same role should continue becoming more present across:

- Chat
- Contacts
- Map
- Calendar
- Reminders
- Wallet
- Shopping
- Food Delivery
- Gallery
- later public/social surfaces

### 2.4 Structured interaction media

The product can grow beyond plain text through:

- richer message cards
- module links
- voice-style interactions
- image-supported interactions
- mini HTML scenes
- future lightweight interactive module surfaces

### 2.5 Public / social / world-facing surfaces

Longer-term directions may include:

- service-account ecosystems
- forum/feed/public-post surfaces
- world-facing creator/configuration tools
- later shared public social-space modules

These should come only after stronger ownership and continuity foundations.

## 3. Current Expansion Constraints

Expansion must respect the current frozen boundaries:

1. `Contacts` is the role-centered archive and management hub.
2. `Chat Directory` is Chat-side binding, not global role truth.
3. `relationship runtime` owns relationship truth.
4. `Calendar` owns confirmed schedule/date meaning.
5. `Reminders` owns raw cues and follow-ups.
6. `Files` stays internal.
7. `World Hub` stays optional and narrow before stronger control lanes mature.
8. Gallery remains asset-first before it becomes a mainline memory-entry surface.
