# Chat And Chat Directory Package

Updated: 2026-06-01

Use this package for Chat thread behavior, Chat Directory, service accounts, role binding into Chat, and message-surface work.

Current 4.4 note: Shopping, Logistics, and Food Delivery can now push reusable Chat service-notification messages with source references and route actions. Chat owns the message history; the source modules keep the business records. Chat-side source notification plans now describe which source modules can push event-driven updates into a joined service account; these plans are descriptive and do not auto-create subscriptions or source records.

Current subscription UX note: service and official accounts now behave more like chat-app subscription channels. Folded subscriptions stay out of Messages but have a visible folded-subscriptions entry back to Services, service threads expose channel status plus mute/fold controls, and Services presents accounts as grouped feed sections (`Unread updates`, `Today`, `Earlier`, `No updates yet`). The folded entry and Services summary now distinguish folded unread, muted unread, and all-read states, so users can tell messages were quietly kept rather than lost. Services also has inbox-style filter explanations, count chips, selected summary cards, and contextual empty states, so users can understand no-unread, muted, folded, search, service-only, and official-only states without reading backend-like terms. When users open a service thread from Services, Chat preserves the source filter on return, highlights the selected service when it is still visible, scrolls it back into view, and shows a recovery panel if the account left the current unread filter because the thread was read. Users can clear unread subscription updates one account at a time or all at once without opening every thread; mark-read feedback explicitly says notification cards stay in the thread. Service threads add date dividers plus a lightweight digest before consecutive notification runs, so repeated pushes read like a message timeline instead of a backend feed. Same-day follow-up pushes from the same service source now use compact cards after the first full card, keeping source-open and Chat reply actions available while making a dense thread readable on one phone screen. Empty service threads now show a subscription-home empty state rather than a blank backend-like surface. Service threads now also show an account-like channel card with the service name, source chips, replyability, source-ownership boundary, history-retention promise, and current inbox placement. Service-notification cards show user-facing source labels instead of backend module keys and separate source-detail actions from Chat reply actions, explaining that source actions open details while replies stay in Chat. Notification cards now also use source-specific presentation for Shopping orders, logistics tracking, and Food Delivery updates so the timeline reads like real chat-app service messages instead of one generic backend feed. Opening a service notification is still a normal chat-thread experience: users can quote/reply to the notification inside Chat while the source module keeps the business record; when users return from a source detail, Chat shows source-open status near the composer, offers an open-again action, and states that Chat did not mutate source-module business state. Service notification reply and sent states also appear near the composer, and opening an unread service thread confirms that Chat only cleared the unread counter while keeping notification cards and source records intact. The composer status, quote bar, errors, and notices now live in normal bottom layout flow rather than overlaying message cards, which reduces mobile action-button overlap. Service-account AI replies receive the service template plus notification context and are instructed to answer conversationally without claiming to mutate source-module business state.

Current WorldBook note: Chat prompt assembly and the thread WorldBook summary now consume the same `world-interface` result, so the user-visible active knowledge state matches the world context sent to AI.

Current World Pack note: active World Pack service-account templates are now surfaced from Current World Pack as availability only, while the actual opt-in lives in Chat Directory's `Services` management area. Users can join enabled world service/official account candidates there; Chat Directory creates or reuses the generated account through the idempotent World Pack service-template seam, records only origin metadata, and keeps WorldBook out of service-account editing/creation. Chat exposes `getServiceAccountLinkContract(contactId)` so follow-up source-module work can inspect the Chat-owned thread route, source bindings, source notification plan, required `service_notification` fields, interaction capabilities, and source-record boundary without reading Chat UI internals.

Current World Pack service-template edit note: built-in active-pack service/official account candidates can now be edited and reset inside Chat Directory's `Services` management area before the user joins them. The editable fields are user-facing account name, account kind, description, and linked world-app binding; user overrides are stored on the World Pack template metadata and marked as customized, while joined Chat accounts remain Chat-owned subscription entries and are not silently overwritten by later template edits. AI-suggested service candidates now enter through the same Chat Services review lane: AI or pasted JSON can propose service/official account templates from active WorldBook/World Pack context, low-confidence/duplicate/unknown-binding proposals are rejected, and confirmed proposals only become World Pack templates until the user manually joins them.

Current Chat App shell note: Chat now presents a messaging-app style product shell with first-level tabs for `Messages`, `Objects`, `Groups`, `Services`, and `Me`. Messages stays the immersive conversation home; object binding, group management, and service accounts have their own first-level tabs. `Me` is the Chat identity/anonymity, saved normal-chat messages, recent interaction data, and derived social-data page, while the top-right gear opens Chat Settings for Chat Appearance, default-behavior entry points, and maintenance diagnostics. Ordinary role/group messages now distinguish delete from recall: delete removes the message without trace, while recall keeps a chat-style notice and tells AI only that the user or contact recalled a message, not the original content. Rich message blocks such as voice, image, transfer, product, link, and mini-scene cards follow the same rule; quote previews and pending quote bars are scrubbed when the original is recalled or deleted. The Chat `+` panel has verified user-send paths for transfer, voice, product, external link, current location, gallery asset, and one-off image/GIF media entry, with copy feedback preserving structured rich-message context. Rich card editing now updates structured fields for link, location, transfer, voice, and image cards so UI, copy, and AI context stay aligned; product and service cards keep source-owned boundaries. `Me` intentionally excludes service-account feed semantics until the World Pack service-account direction settles.

Current group-chat note: group chats are first-class Chat targets backed by Chat-side role member ids and a group reply mode. V1 stores and surfaces the group target cleanly; deeper multi-speaker orchestration remains a future behavior layer.

Current relationship-compatibility note: Chat Directory still preserves legacy `relationshipLevel` and `relationshipNote` fields for binding compatibility, but the UI labels them as Chat-local tuning/note only. Current relationship truth stays in relationship runtime.

Incoming social-event note: Chat social/channel state has a V1 applied shell for direct user actions such as greetings, request acceptance/decline, blocking, unblocking, and blocked-by-role markers. Generated role-initiated social events now enter the explicit review seam: Chat owns messaging accessibility and visible social-channel state, Contacts displays the role-level snapshot only, Event Runtime owns eligibility/review/audit, World Hub reviews high-risk proposals, and relationship runtime receives only confirmed relationship facts or memories.
